# 筛选结果无法全选 根因分析

## 1. 问题定位

| 关键位置 | 说明 |
|---|---|
| `frontend/src/history-workspace-thread-table.tsx:32` | 表头勾选框被设为只读，只显示“已有选中项”，没有全选事件。 |
| `frontend/src/history-workspace-controller-internals.ts:220` | 选择动作只提供逐条切换和选择建议项，没有切换全部可见会话的动作。 |
| `frontend/src/history-workspace-contract.ts:74` | 控制器契约没有暴露全选当前筛选结果的能力。 |

## 2. 失败路径还原

**正常路径**：用户筛选会话后逐条点击行勾选框，前端更新手动选择列表，删除计划使用当前选中会话。

**失败路径**：用户筛选会话后点击表头勾选框，表头输入框没有事件处理且为只读，选择状态不发生变化。

**分叉点**：`frontend/src/history-workspace-thread-table.tsx:32` — 表头控件只有状态展示，没有交互行为。

## 3. 根因

**根因类型**：逻辑错误。

**根因描述**：界面使用了通常代表“全选”的表头勾选框，但实现仅把它当成只读状态标记。选择控制器也没有对应的“选择或取消当前筛选结果”动作，因此筛选后无法批量选中。

**是否有多个根因**：否。控制器缺少动作和表头只读是同一处能力没有接通的两端表现。

## 4. 影响面

- **影响范围**：所有筛选组合都无法全选，不只影响“一个月之前”。
- **潜在受害模块**：会话列表选择、删除计划入口和选中数量展示；删除执行逻辑本身不受影响。
- **数据完整性风险**：无。当前问题不会误删数据，修复后仍经过删除预览和二次确认。
- **严重程度复核**：维持 P1。核心批量操作受损，但可以逐条勾选绕过。

## 5. 修复方案

### 方案 A：让现有表头勾选框支持全选

- **做什么**：增加“切换全部可见会话”的选择动作；表头勾选框根据当前筛选结果显示未选、部分选、全选三种状态。
- **优点**：符合表格通用交互，入口现成，改动集中且不增加新界面元素。
- **缺点 / 风险**：需要正确处理部分选中和筛选变化后的状态。
- **影响面**：前端选择控制器、控制器契约和会话表格。

### 方案 B：增加独立的“全选筛选结果”按钮

- **做什么**：在列表工具栏增加全选和取消选择按钮，继续保留只读表头。
- **优点**：文案可以明确说明作用范围。
- **缺点 / 风险**：保留一个看似可点击但实际只读的表头勾选框，界面更拥挤，用户仍可能困惑。
- **影响面**：前端选择控制器、控制器契约和列表工具栏。

### 推荐方案

**推荐方案 A**，理由：直接修复误导性的现有控件，交互符合用户预期，且不会改变删除安全流程。用户已提前确认文档完成后直接修复，因此本方案视为已确认。
